关键设计亮点：
- **get_sdk_reference** 返回 SDK 文档（patterns/expressions/functions/rules/import/guidelines/design），构建工作流前必须先调用 [web-reader]

### 1.2 czlonkowski/n8n-mcp（社区第三方，最成熟）

由 @czlonkowski 开发的独立 MCP server，是 n8n 生态中最受欢迎的第三方 MCP 服务。MIT 开源，5418 个测试通过。[web-reader] ✅ 双路验证（GitHub README + n8n-mcp.com）

## 二、信息架构：n8n MCP Server 的工具设计

### 2.1 n8n 官方 MCP 的工具分组逻辑

n8n 官方 MCP 采用**三层分组**：[web-reader]

```
Workflow Management（生命周期管理）
  ├── 搜索/查看：search_workflows, get_workflow_details
  ├── 执行/测试：execute_workflow, get_execution, test_workflow, prepare_test_pin_data
  ├── 发布/下线：publish_workflow, unpublish_workflow
  └── 组织管理：search_projects, search_folders

Workflow Builder（开发辅助）
  ├── 文档参考：get_sdk_reference
  ├── 节点发现：search_nodes, get_node_types, get_suggested_nodes
  ├── 校验：validate_workflow
  └── 代码生成：create_workflow_from_code, update_workflow, archive_workflow

Data Tables（数据管理）
  └── CRUD：search/create/rename/delete/操作列/行
```

### 2.2 每个工具的输入输出规范

以 n8n 官方 MCP 为例，工具设计遵循以下规范：[web-reader]

**输入规范：**
- 必填参数用 `required: true` 标记
- 类型约束明确：string/integer/boolean/array/object + enum
- 支持 discriminated union（如 execute_workflow 的 inputs 字段按 type 区分 chat/form/webhook）
- 默认值明确声明

**输出规范：**
- 统一返回结构化 JSON 对象
- 成功时含 `success: boolean` + 业务数据
- 失败时含 `error: string`（人类可读错误信息）
- 列表类接口含 `count: integer` 总数
- 时间字段统一 ISO 格式
- 敏感信息自动脱敏（如 credentials）

### 2.3 czlonkowski/n8n-mcp 的分组逻辑

社区版采用**两层分组**，更扁平：[web-reader]

```
Core Tools（7 个，无状态，纯文档/校验）
  ├── 文档：tools_documentation
  ├── 节点发现：search_nodes, get_node（多模式）
  ├── 校验：validate_node（两级）, validate_workflow
  └── 模板：search_templates, get_template

Management Tools（13 个，有状态，需 API）
  ├── 工作流 CRUD + 版本 + 部署
  ├── 执行管理
  ├── 凭据管理
  ├── 安全审计
  └── 系统健康检查
```

## 三、交互模式：Claude Code 调用 MCP tool 的典型工作流

### 3.1 n8n 官方推荐的开发流程

根据 n8n 官方文档，AI 辅助开发工作流的标准流程为：[web-reader]

```
1. get_sdk_reference()          → 先获取 SDK 文档
2. search_nodes(queries)        → 搜索需要的节点
3. get_node_types(nodeIds)      → 获取节点类型定义
4. get_suggested_nodes(categories) → 获取推荐节点（可选）
5. AI 生成 SDK 代码
6. validate_workflow(code)      → 校验代码
7. create_workflow_from_code()  → 创建工作流
8. publish_workflow()           → 发布上线
```

### 3.2 czlonkowski/n8n-mcp 推荐的开发流程

社区版提供更详细的多级校验流程：[web-reader]

```
1. tools_documentation()                              → 获取最佳实践
2. search_templates(searchMode, ...)                  → 先查模板（Templates First 原则）
3. search_nodes(query, includeExamples: true)         → 搜索节点（并行）
4. get_node(nodeType, detail: 'standard')             → 获取节点配置
5. validate_node(nodeType, config, mode: 'minimal')   → 快速校验
6. validate_node(nodeType, config, mode: 'full')      → 完整校验
7. 构建工作流 JSON
8. validate_workflow(workflow)                        → 工作流级校验
9. n8n_create_workflow(workflow)                      → 部署
10. n8n_validate_workflow({id})                       → 部署后校验
11. n8n_test_workflow({workflowId})                   → 测试执行
```

### 3.3 文档查询 vs 校验的调用时机

| 阶段 | 调用的工具类型 | 目的 |
|------|--------------|------|
| 需求理解 | 文档查询（get_sdk_reference, tools_documentation） | 了解平台能力和约束 |
| 节点选择 | 搜索（search_nodes, search_templates） | 找到合适的节点和模板 |
| 配置编写 | Schema 查询（get_node_types, get_node） | 获取准确的参数定义 |
| 代码生成 | 无（AI 生成） | 生成工作流 DSL |
| 校验修复 | 校验（validate_node, validate_workflow） | 检查错误并修复 |
| 部署测试 | 管理（create, publish, test） | 部署和验证 |

## 四、AI 化案例

### 4.1 n8n-mcp 的端到端案例

n8n-mcp 官网展示了完整的「自然语言 → 工作流 → 部署」流程：[web-reader]

1. 用户在 Chat Agent 中描述需求："When a new GitHub issue is created, send a Slack notification"
2. AI 调用 `search_templates` 查找匹配模板
3. AI 调用 `search_nodes` 发现 GitHub Trigger 和 Slack 节点
4. AI 调用 `get_node` 获取节点配置详情
5. AI 生成工作流代码
6. AI 调用 `validate_workflow` 校验
7. AI 调用 `n8n_create_workflow` 部署
8. AI 调用 `n8n_test_workflow` 测试

整个过程可在 60 秒内完成。[web-reader] n8n-mcp.com

## 五、推送/触达形态：MCP 返回结果格式最佳实践

### 5.1 MCP 协议规范

MCP 协议规范（draft）定义了 tool 返回格式：[web-search-prime]

- **结构化内容**：通过 `structuredContent` 字段返回 JSON 值（object/array/string/number/boolean）
- **文本内容**：通过 `content` 字段返回 text/image/resource 类型
- 工具可以同时返回结构化内容和文本内容

### 5.2 最佳实践（Philipp Schmid 总结）

Philipp Schmid 在 2026-01 的博文中总结了 MCP Server 构建最佳实践：[web-reader]

**返回格式设计原则：**

1. **结果导向，非原始数据**：不要返回 API 原始响应，返回 AI 能直接理解的结果
   - 错误：`{"messages": [{"id": "123", "threadId": "abc", "payload": {"headers": [...], "body": {"data": "base64..."}}}]`
   - 正确：`{"subject": "...", "sender": "...", "body": "...", "attachments": ["file.pdf"]}`

2. **扁平化参数**：避免嵌套字典，使用顶层基本类型和约束类型
   - 错误：`search_orders(filters: dict)`
   - 正确：`search_orders(email: str, status: Literal["pending", "shipped", "delivered"], limit: int = 10)`

3. **指令即上下文**：docstring 和错误信息都是 AI 的上下文
   - 错误：抛出 Python 异常
   - 正确：返回 `"User not found. Please try searching by email address instead."`

4. **精简返回**：分页返回大结果集
   - 支持 `limit` 参数（默认 20-50）
   - 返回 `has_more`, `next_offset`, `total_count` 元数据

5. **工具命名**：`{service}_{action}_{resource}` 格式，避免冲突

[web-reader] philschmid.de/mcp-best-practices

### 5.3 n8n MCP 的返回格式实践

n8n 官方 MCP 的返回格式设计值得参考：[web-reader]

- **成功响应**：`{"success": true, "workflowId": "...", "url": "...", "nodeCount": 5}`
- **校验响应**：`{"valid": true, "nodeCount": 3, "warnings": [...], "errors": [...]}`
- **列表响应**：`{"data": [...], "count": 10}`
- **错误响应**：`{"error": "Human-readable error message"}`
- **文档响应**：`{"reference": "Markdown formatted documentation text"}`

### 5.4 对 Dify 工作流 MCP 服务的启示

基于以上调研，Dify 工作流 MCP 服务的返回格式建议：

| 工具类型 | 推荐格式 | 理由 |
|---------|---------|------|
| 文档查询 | Markdown 文本 | AI 直接理解，无需解析 |
| 节点 Schema | 结构化 JSON | 需要精确的类型信息 |
| 变量引用 | 结构化 JSON + 示例 | AI 需要准确的路径和类型 |
| 校验结果 | 结构化 JSON（valid/errors/warnings） | AI 需要程序化处理错误 |
| DSL 生成 | JSON（Dify DSL 格式） | 直接可用的工作流定义 |

## 六、产品形态小结

### 核心发现

1. **市场空白确认**：在「AI 辅助工作流开发」MCP 赛道，n8n 是唯一有成熟实现的平台。Dify、Coze、FastGPT 均无类似产品。这是一个明确的产品机会。[web-search-prime] ⚠️ 单源

2. **n8n 是最佳参照物**：n8n 的官方 MCP + czlonkowski/n8n-mcp 社区版共同构成了最完整的参考实现，涵盖文档查询、节点发现、Schema 获取、多级校验、代码生成、部署测试全链路。[web-reader] ✅ 双路验证

3. **工具数量控制在 5-15 个**：根据 MCP 最佳实践，单个 MCP Server 应暴露 5-15 个工具。n8n 官方 20+ 个偏多，但分为三组后每组 6-10 个合理。Dify 工作流 MCP 服务应精简工具数。[web-reader]

4. **返回格式以结构化 JSON 为主**：校验结果、节点 Schema 等需要程序化处理的用 JSON；文档查询用 Markdown；错误信息要人类可读、可操作。[web-reader]

5. **多级校验是核心差异点**：n8n 的 validate_node（minimal/full）→ validate_workflow 多级校验模式是其最大亮点，Dify 工作流 MCP 服务应实现类似的 DSL 校验能力。[web-reader]

6. **模板先行策略有效**：czlonkowski/n8n-mcp 的「Templates First」原则（先搜模板再从零构建）显著提高了 AI 生成工作流的质量和速度。Dify 应考虑内置 DSL 模板库。[web-reader]

7. **Dify 的独特优势**：Dify 工作流使用 YAML DSL 而非代码，校验可以更精确；Dify 的节点类型相对 n8n 更少（约 20+ 种 vs 1851 种），文档维护成本更低。[web-search-prime] ⚠️ 单源

### 推荐产品形态

基于调研结论，Dify 工作流 MCP 服务应定位为：**面向 Claude Code 编程智能体的 Dify 工作流开发辅助工具**，核心能力包括：

- **文档查询**：Dify 工作流节点文档、变量引用语法、DSL 规范
- **节点 Schema**：每种节点类型的完整参数定义和约束
- **DSL 校验**：多级校验（语法 → 结构 → 语义 → 连通性）
- **模板库**：内置常用工作流模板，支持搜索和获取
- **API 集成**：工作流的创建、更新、发布、测试（可选，MVP 后实现）

工具数量建议控制在 **8-12 个**，分两组：
- Core Tools（6-8 个）：文档查询、节点发现、Schema 获取、DSL 校验、模板搜索
- Management Tools（2-4 个，可选）：工作流 CRUD、发布测试
