## 1. 决策摘要

**架构基线**：**Mintlify MCP + yzmw123 Skill 适配**——用 Mintlify 原生 MCP 免费覆盖文档查询，基于 yzmw123/dify-workflow-dsl-skill 适配 Skill 补充 Schema/校验/模板。按需升级为折中 MCP Server。

**一句话理由**：Mintlify MCP 免费覆盖文档查询；yzmw123 已覆盖 80%+ Skill 需求（25+ Schema、校验脚本、4 模板），Phase 1 仅需 0.5-0.75 天适配工作。

**三层架构**（详见 `07-项目架构.md`）：
- Layer 1：Mintlify MCP → 文档搜索 + 页面读取（免费）
- Layer 2：yzmw123 Skill 适配 → Schema + 校验 + 模板（0.5-0.75 天）
- Layer 3：自建 MCP Server → 离线 + 结构化 + API 集成（按需 10-15 天）

**验证结果**：Mintlify MCP 已确认可用 ✅
- 端点：`https://dify-6c0370d8.main-kill-isr.mintlify.me/mcp`
- 零认证、Streamable HTTP、5,000 请求/小时
- 2 个工具：`search_dify_docs`（语义搜索）+ `query_docs_filesystem_dify_docs`（虚拟文件系统，支持 rg/cat/tree）
- 覆盖 Dify v1.9.x ~ v1.10.x，21 个节点文档页面，文档质量极好

## 2. 判决过程

### 2.1 幸存论点分析

**被驳倒的论点**（不计入判决依据）：

| 论点 | 提出者 | 驳倒方 | 驳倒理由 |
|------|--------|--------|---------|
| "200 页文档需要 FAISS+FTS5 混合检索" | 代言人 A | 红队质疑 1.2 + 集成评估师 | Claude Code 200K context 可装入全部文档；n8n-mcp 1851 节点用纯 FTS5 验证过；100-200 页规模下 FTS5 与向量检索差距有限 |
| "graphon 运行时依赖可接受" | 代言人 A | 红队质疑 1.1 | graphon 0.4.0 "still evolving"，22-30 天项目建在 0.x 版本包上是工程赌博。代言人 C 的"一次性提取"方案更好 |
| "Skill 的结构化查询缺陷不可接受" | 红队质疑 2.2 | 代言人 B | 当前阶段目标是验证需求是否存在，而非追求完美体验。Skill 的 80% 覆盖率对 MVP 足够 |

### 2.2 判决逻辑

**第一步：否决方案 1 直接执行**

22-30 人天的全功能 MCP Server 在需求未验证阶段投入过高。红队的核心论点——"为 200 页文档建 RAG 系统是过度工程"——未被有效反驳。graphon 0.4.0 的不稳定性风险真实存在。官方 dify-docs-mcp-server 5 个月归档的事实进一步削弱了"必须自建"的论证。

**第二步：否决"纯 Skill 长期使用"**

Skill 的功能天花板是真实的：无法做程序化 DSL 校验、无法集成 Dify API、仅支持 Claude Code。集成评估师给出的扩展性评分 2/10 准确反映了这一现实。但 Skill 作为 MVP 验证工具是合理的。

**第三步：选择渐进路径**

集成评估师的评分最能指导决策：
- A→C 路径（先折中 MCP）：8.5 分
- B→A 路径（先 Skill 后按需 MCP）：8.0 分

两者差距不大，但 B→A 路径的风险更低（1-2 天 vs 10-15 天的初始投入），且 Skill 的知识资产可复用到 MCP Server。在需求未验证的前提下，**风险最低的路径是最理性的选择**。

**第四步：加入 Mintlify MCP 验证前置步骤**

红队提出的方案 4（Mintlify 原生 MCP）是一个被忽视的重要选项。如果 Mintlify 已为 docs.dify.ai 提供原生 MCP Server，文档查询部分可以零成本实现。这应该在做 Skill 之前优先验证。

## 3. 推荐执行路径（v3）

```
Phase 0a（已完成）：验证 Mintlify MCP ✅
  └─ 端点：https://dify-6c0370d8.main-kill-isr.mintlify.me/mcp
  └─ 2 个工具：search_dify_docs + query_docs_filesystem_dify_docs
  └─ 配置已写入 .mcp.json

Phase 0b（已完成）：验证开源 Skill 项目 ✅
  └─ yzmw123/dify-workflow-dsl-skill 覆盖 80%+ 需求
  └─ r-hashi01 提供节点输出字段参考
  └─ R3flector 提供 8 个工作流模式

Phase 1（0.5-0.75 天）：Skill 适配（基于 yzmw123）
  └─ 适配 yzmw123 的 SKILL.md + references/（基座，直接适配）
  └─ 补充 rag 变量前缀文档（0.5h，源码已确认）
  └─ 移植节点输出字段参考（1h，从 r-hashi01）
  └─ 补充 2-4 个工作流模板（1-2h，从 R3flector）
  └─ 新增 Mintlify MCP 使用指引（0.5h）
  └─ 集成测试：生成 3-5 个实际工作流（1-2h）
  └─ 文档查询由 Mintlify MCP 覆盖，不在 Skill 范围内
  └─ 团队内部试用 2-4 周

Phase 2（按需，10-15 天）：折中 MCP Server
  └─ 触发条件：团队反馈不够用（见下方判断标准）
  └─ FastMCP 框架 + SQLite FTS5 + 本地 JSON Schema
  └─ 8 个 MCP Tools
  └─ 可选升级：+FAISS 向量检索（3-5 天）
```

### Phase 1 → Phase 2 的判断标准

如果团队在试用 Mintlify MCP + Skill 期间遇到以下情况，启动 Phase 2：

1. Mintlify MCP 文档查询不够精准或延迟太高（需本地 FTS5 替代）
2. Skill 中的 Schema 信息不够精确（需结构化 JSON 替代 Markdown）
3. 需要程序化 DSL 校验（Skill 的 checklist 不够精确）
4. 需要直接从 Claude Code 执行 Dify 工作流
5. 团队中有人用 Cursor/Windsurf，需要 MCP 协议支持
6. Skill 内容占用过多 context window

## 5. 被否方案理由

### 方案 1（全功能 MCP Server）为什么不做首选

被以下论点驳倒：
- **红队质疑 1.2**：200 页文档不值得建 RAG 系统。Claude Code 200K context 可装入全部文档纯文本（约 50-100K token），FAISS+FTS5 混合检索的投入产出比不合理。
- **红队质疑 1.3**：官方 dify-docs-mcp-server 5 个月就归档（4 commits, 8 stars），说明 Dify 官方自己都认为"为文档建 MCP Server"不值得持续投入。
- **红队质疑 1.1**：graphon 0.4.0 "still evolving"，22-30 天项目建在 0.x 版本的第三方包上是工程赌博。
- **集成评估师评分**：风险度 4/10，功能覆盖度 9/10 但开发成本 3/10，综合 6.2 分。

**注意**：方案 1 不是被永久否决，而是被推迟到需求验证之后。如果 Phase 1/2 验证了需求确实存在，方案 1 仍然是功能最完整的选项。

### 纯 Skill 长期使用为什么不行

- **集成评估师评分**：扩展性 2/10，功能覆盖度 4/10
- **红队质疑 2.2**：纯 Markdown 无法提供结构化查询
- **红队质疑 2.3**：2-4 周 Schema 过期

### "全自建"为什么不行

- 红队自己的评估：24-34 人天，比方案 1 更贵更慢
- 根本问题相同：为 200 页文档建 RAG 系统是过度工程
