## 1. 单方案评分表

评分标准：1 分（极差）~ 10 分（极优），6 分为及格线。

| 维度 | 方案 1：全功能 MCP Server | 方案 2：极简 Skill | 方案 3：折中 MCP Server |
|------|--------------------------|-------------------|------------------------|
| **功能覆盖度** | **9** | **4** | **7** |
| **开发成本** | **3** | **9** | **6** |
| **维护成本** | **4** | **8** | **6** |
| **用户体验** | **8** | **5** | **7** |
| **扩展性** | **9** | **2** | **7** |
| **风险度** | **4** | **7** | **6** |
| **综合分** | **6.2** | **5.8** | **6.5** |

### 1.1 方案 1：全功能 MCP Server（综合 6.2）

**功能覆盖度（9/10）**：覆盖全部 8 个工具（search_docs / list_node_types / get_node_schema / validate_dsl / search_templates / get_template / run_workflow / get_workflow_run），文档查询有 FAISS+FTS5 混合检索，DSL 校验有 jsonschema+语义规则引擎。是三方案中功能最完整的。

**开发成本（3/10）**：22-30 人天，约 5-7 周 1 人全职。涉及 FastMCP 框架学习、FAISS 向量索引搭建、SQLite FTS5 全文索引、graphon Pydantic model 解析、JSON Schema 逆向构建、DSL 语义规则引擎等多个子系统。学习曲线陡峭，尤其 Embedding 配置和 FAISS 调优。

**维护成本（4/10）**：依赖链长且脆弱——graphon 包"still evolving"（0.4.0），Dify 文档无版本号（URL 中无版本标识），Embedding API 有网络依赖和计费，FAISS 索引需定期重建。文档更新时需重跑抓取+分块+向量化流程。graphon 版本升级可能导致 entities.py 结构变化，Schema 提取脚本需同步维护。

**用户体验（8/10）**：Claude Code 通过 stdio JSON-RPC 调用结构化工具，响应速度快（本地 SQLite 查询 <100ms），返回格式规范（JSON Schema + Markdown）。8 个工具覆盖完整开发流程。

**扩展性（9/10）**：MCP 框架天然支持后续加工具，FastMCP 装饰器式 API 添加新 tool 只需一个函数。可扩展 SSE/HTTP 模式支持远程访问，可加 Docker 部署。

**风险度（4/10）**：多层风险叠加——(1) graphon 包 0.4.0 "still evolving"，entities.py 结构可能变化；(2) 官方 dify-docs-mcp-server 已归档（存活仅 5 个月），说明 Dify 团队对文档 MCP 的态度不明朗；(3) FAISS+Embedding 配置对团队技术栈有要求；(4) 22-30 人天的投入在需求未验证阶段风险较高。

### 1.2 方案 2：极简 Skill（综合 5.8）

**功能覆盖度（4/10）**：仅覆盖文档查询和基础 Schema 查询两个维度。通过 llms.txt 实时抓取获取文档，一次性提取 Schema 内嵌到 Markdown 中。无法做 DSL 校验（Skill 无法运行代码）、无法集成 Dify API（无 HTTP 客户端）、无法做混合检索（纯文本匹配）。

**开发成本（9/10）**：1-2 人天。编写 Markdown 格式的 Skill 文件，包含系统提示词、Schema 知识、变量语法规则。参考 yzmw123/dify-workflow-dsl-skill 已有先例，可快速复制模式。

**维护成本（8/10）**：极低。无外部依赖、无索引、无服务进程。更新只需编辑 Markdown 文件。Schema 变化时手动更新内嵌内容即可。唯一的维护成本是文档/Schema 变化时需要人工同步更新 Skill 内容。

**用户体验（5/10）**：Claude Code 通过 Skill 系统加载 Markdown 内容到上下文。问题在于：(1) Skill 内容会占用 context window，文档量大时 token 消耗高；(2) 查询不精准——LLM 需要从大量上下文中自行筛选相关内容，无法像 MCP tool 那样精确返回；(3) 无法做实时检索，所有知识都是静态的。

**扩展性（2/10）**：几乎无法扩展。Skill 是纯 Markdown，无法添加代码逻辑。如果后续需要 DSL 校验、API 集成、向量检索等功能，必须完全重写为 MCP Server。Skill 和 MCP Server 之间没有渐进升级路径——是两种完全不同的技术形态。

**风险度（7/10）**：风险低但有天花板。(1) 功能天花板明显——无法做校验和 API 集成；(2) 如果用户反馈"不够用"，升级到 MCP Server 的成本等于从零开始；(3) llms.txt 实时抓取依赖网络，中国大陆网络环境可能影响稳定性。

### 1.3 方案 3：折中 MCP Server（综合 6.5）

**功能覆盖度（7/10）**：覆盖文档查询（llms.txt 轻量检索，无向量索引）、Schema 查询（一次性提取本地维护）、基础校验（jsonschema 结构校验，无语义规则）。相比方案 1 缺少 FAISS 语义检索和完整语义校验，相比方案 2 多了 MCP 框架和基础校验能力。

**开发成本（6/10）**：10-15 人天，约 2-3 周。比方案 1 少 10-15 人天，主要省在：(1) 不建 FAISS 向量索引（省 3-5 天）；(2) 不做 Embedding 集成（省 1-2 天）；(3) 不依赖 graphon 运行时（省 2-3 天，改为一次性提取）。但仍需 FastMCP 框架搭建、SQLite FTS5 全文索引、Schema 数据整理。

**维护成本（6/10）**：中等。不依赖 Embedding API 和 FAISS 索引，减少了运行时依赖。但 llms.txt 抓取逻辑需维护（Dify 文档结构变化时需更新），Schema 提取为一次性但 Dify 版本升级时需手动更新。SQLite FTS5 为标准库，维护成本低。

**用户体验（7/10）**：有 MCP Server 的结构化工具调用体验，但检索能力不如方案 1。对于 100-200 页的 Dify 文档，FTS5 全文检索基本够用，但对模糊语义查询（如"怎么实现条件分支"）效果不如向量检索。

**扩展性（7/10）**：MCP 框架支持后续加工具。关键扩展路径清晰：验证需求后可加 FAISS 向量检索（方案 3 → 方案 1 的升级路径），也可加 graphon 运行时依赖增强 Schema 管理。架构设计为渐进增强留了空间。

**风险度（6/10）**：风险可控。(1) 不依赖 graphon 运行时，降低了一层依赖风险；(2) 不建向量索引，降低了技术复杂度风险；(3) 但 MCP Server 仍需 Python 环境和 pip 依赖，比 Skill 部署复杂；(4) llms.txt 实时抓取的网络依赖仍在。

## 2. 组合方案评估表

| 组合路径 | 路径描述 | 验证周期 | 总投入 | 风险等级 | 适用场景 |
|---------|---------|---------|--------|---------|---------|
| **A→B** | 先做 Skill（MVP）→ 验证需求 → 再做 MCP Server | 2-4 周验证 + 5-7 周开发 | 23-32 人天 | 低 | 需求高度不确定，需要低成本试错 |
| **A→C** | 先做折中 MCP → 验证 → 再加向量检索 | 2-3 周验证 + 3-5 天升级 | 13-20 人天 | 中低 | 需求基本确定，追求务实渐进 |
| **B→A** | 先做 Skill → 用户反馈不够 → 升级 MCP | 1-2 天验证 + 22-30 天开发 | 23-32 人天 | 低 | 最稳妥的渐进式路径 |
| **纯 B** | 一直用 Skill，不做 MCP | 1-2 天 | 1-2 人天 | 中 | 预算极紧或仅需快速 PoC |

### 2.1 A→B 路径：先 Skill 后 MCP（推荐度：7/10）

**流程**：1-2 天做 Skill → 团队内部试用 2-4 周 → 收集反馈 → 若不够用则做 MCP Server

**优势**：
- 低成本验证核心假设——"Claude Code 开发 Dify 工作流时是否真的需要文档/Schema 辅助"
- Skill 的 1-2 天投入几乎零风险
- 用户反馈可直接指导 MCP Server 的工具设计优先级

**劣势**：
- 验证周期长（2-4 周才能收集到有意义的反馈）
- Skill 和 MCP Server 是不同技术形态，Skill 的代码/架构无法复用到 MCP Server
- 如果 Skill 验证"够用"，可能误导决策——Skill 的"够用"可能是团队规模小、使用频率低导致的假象

**关键判断点**：如果团队在试用 Skill 期间频繁遇到以下问题，说明需要升级 MCP Server：
1. 文档查询不精准，LLM 从大量上下文中找不到答案
2. 需要 DSL 校验能力（Skill 无法提供）
3. 需要直接调用 Dify API 执行工作流
4. Skill 内容占用过多 context window

### 2.2 A→C 路径：先折中 MCP 后加向量检索（推荐度：8/10）

**流程**：10-15 天做折中 MCP Server → 团队试用 2-3 周 → 若检索不够精准则加 FAISS 向量检索

**优势**：
- 起步就有 MCP Server 的完整框架体验（结构化工具、stdio 集成）
- 不建向量索引降低了初始复杂度，10-15 天可交付可用产品
- 升级路径清晰：加 FAISS+Embedding 模块即可升级为方案 1
- 对于 100-200 页的 Dify 文档，FTS5 全文检索可能已经够用，不需要向量检索

**劣势**：
- 10-15 天的投入在需求未验证阶段仍有一定风险
- 折中 MCP 的检索能力（纯全文）可能不够用，导致用户体验不佳
- 如果最终发现需要向量检索，总投入接近方案 1

**关键判断点**：以下场景测试折中 MCP 的检索能力：
1. 精确查询（如"llm 节点的 model 参数"）——FTS5 应该能处理
2. 模糊查询（如"怎么实现循环"）——FTS5 可能不够，需要向量检索
3. 跨文档查询（如"变量引用和节点配置的关系"）——需要语义理解

### 2.3 B→A 路径：先 Skill 后按需升级 MCP（推荐度：8/10）

**流程**：1-2 天做 Skill → 团队试用 → 若反馈不够 → 22-30 天做全功能 MCP Server

**优势**：
- 最稳妥的渐进式路径，每一步都有明确的决策点
- Skill 的知识内容（Schema 定义、变量语法规则、文档摘要）可直接复用到 MCP Server 的数据层
- 如果 Skill 验证"够用"，直接省了 22-30 人天的 MCP Server 开发

**劣势**：
- 如果最终需要 MCP Server，总周期最长（1-2 天 + 验证期 + 22-30 天）
- Skill 验证期间团队体验较差（文档查询不精准、无校验能力）
- Skill → MCP Server 的过渡期可能出现"Skill 已不够用但 MCP Server 还没好"的尴尬

**关键判断点**：与 A→B 路径类似，但更强调"Skill 的知识资产复用"——Skill 中的 Schema 定义、变量语法规则、文档摘要可以直接提取为 MCP Server 的 SQLite 数据。

### 2.4 纯 B 路径：一直用 Skill（推荐度：4/10）

**流程**：1-2 天做 Skill → 长期使用

**优势**：
- 成本最低，1-2 天即可交付
- 零维护成本，无外部依赖
- 适合"只需要文档查询，不需要校验和 API"的极简场景

**劣势**：
- 功能天花板明显——无法做 DSL 校验、无法集成 Dify API
- 扩展性为零——Skill 是纯 Markdown，无法添加代码逻辑
- 如果团队规模扩大或使用频率增加，Skill 的体验瓶颈会越来越明显
- 与 n8n 生态的 MCP Server 相比，功能差距过大，可能影响团队对工具的信心

**适用场景**：仅推荐用于以下情况：
- 预算极紧（<2 人天）
- 仅需快速 PoC 验证"AI 辅助 Dify 工作流开发"这个概念
- 团队规模极小（1-2 人），使用频率极低

| 排名 | 方案/路径 | 推荐分 | 理由 |

| **1** | **A→C 路径**：先折中 MCP → 验证 → 加向量检索 | **8.5** | 务实渐进，起步就有 MCP 框架体验，升级路径清晰，总投入可控（13-20 人天）。对 100-200 页文档 FTS5 可能够用，避免过度工程化 |

| **3** | **方案 3 单独执行**：折中 MCP Server | **7.5** | 10-15 人天可交付可用产品，功能覆盖度和开发成本的最佳平衡点。即使不升级到方案 1，也能满足大部分需求 |

### Top 3 推荐

**Top 1：A→C 路径（先折中 MCP → 验证 → 加向量检索）—— 8.5 分**

核心理由：这是"工程务实主义"的最佳体现。折中 MCP Server 在 10-15 天内可交付一个有完整 MCP 框架体验的产品，覆盖文档查询（FTS5）、Schema 查询、基础校验三大核心能力。对于 100-200 页的 Dify 文档，FTS5 全文检索大概率够用，不需要一开始就引入 FAISS+Embedding 的复杂度。升级路径清晰——如果后续发现检索不够精准，加 FAISS 模块即可。总投入 13-20 人天，是三方案中功能/成本比最优的。

**Top 2：B→A 路径（先 Skill → 按需升级 MCP）—— 8.0 分**

核心理由：最稳妥的渐进式路径。1-2 天的 Skill 投入几乎零风险，团队可以立即开始使用。Skill 中积累的 Schema 知识、变量语法规则、文档摘要等资产，后续可直接复用到 MCP Server 的数据层。关键优势是"每一步都有明确的决策点"——如果 Skill 够用就省了 22-30 人天，如果不够用也有清晰的升级路径。唯一劣势是总周期可能最长。

**Top 3：方案 3 单独执行（折中 MCP Server）—— 7.5 分**

核心理由：如果团队对"AI 辅助 Dify 工作流开发"这个需求已有较高信心（非 PoC 阶段），直接做折中 MCP Server 是最高效的选择。10-15 人天可交付一个有 MCP 框架体验、覆盖核心功能的产品。不建向量索引、不依赖 graphon 运行时，大幅降低了技术风险和维护成本。即使不升级到方案 1，FTS5 全文检索+本地 Schema 也能满足 80% 的使用场景。

## 4. 关键建议

### 4.1 决策框架：三个关键问题

在选择方案之前，建议团队先回答以下三个问题：

| 问题 | 如果是 | 如果否 |
|------|--------|--------|
| Q1：团队是否确信"AI 辅助 Dify 工作流开发"是刚需？ | 跳过 Skill，直接做 MCP Server（方案 3） | 先做 Skill 验证（B→A 路径） |
| Q2：100-200 页文档的检索，FTS5 全文搜索是否够用？ | 折中 MCP 即可（方案 3） | 需要向量检索（方案 1） |
| Q3：团队是否有 2-3 周的连续开发时间？ | 直接做 MCP Server（方案 3 或 1） | 先做 Skill 过渡（B→A 路径） |

### 4.2 不推荐的路径

**不推荐直接做方案 1（全功能 MCP Server）**：22-30 人天的投入在需求未验证阶段风险过高。graphon 包"still evolving"（0.4.0），FAISS+Embedding 配置复杂，依赖链长且脆弱。建议先用方案 3 验证需求，再按需升级。

**不推荐纯 B 路径（一直用 Skill）**：Skill 的功能天花板明显（无法做校验和 API 集成），扩展性为零。除非团队只需要"查文档"这一个功能，否则 Skill 不是长期方案。

### 4.3 方案 3 的关键简化点

如果选择方案 3（折中 MCP Server），以下是建议的简化点：

| 简化项 | 方案 1 做法 | 方案 3 简化做法 | 省多少 |
|--------|-----------|---------------|--------|
| 文档检索 | FAISS + Embedding + FTS5 混合 | 纯 SQLite FTS5 全文检索 | 省 3-5 天 |
| Schema 来源 | graphon 运行时依赖 + 动态解析 | 一次性提取，静态 JSON 文件 | 省 2-3 天 |
| DSL 校验 | jsonschema + 语义规则引擎 | 仅 jsonschema 结构校验 | 省 2 天 |
| Embedding | 可配置 Embedding API | 不需要 | 省 1-2 天 |

### 4.4 方案 3 的升级触发条件

如果团队在使用折中 MCP Server 期间遇到以下情况，说明需要升级到方案 1：

1. **检索不够精准**：用户反馈"搜不到想要的内容"超过 30% 的查询——加 FAISS 向量检索
2. **Schema 过期**：Dify 版本升级后 Schema 不准——加 graphon 运行时依赖做动态解析
3. **校验不够深入**：用户反馈"校验通过但导入失败"——加语义规则引擎
4. **需要 API 集成**：用户需要直接从 Claude Code 执行 Dify 工作流——加 httpx + run_workflow tool

### 4.5 通用注意事项

1. **llms.txt 抓取的网络依赖**：中国大陆网络环境访问 docs.dify.ai 可能不稳定，建议实现本地缓存+离线模式
2. **graphon 包版本锁定**：无论选择哪个方案，都应锁定 graphon 版本（`graphon==0.4.0`），避免"still evolving"带来的 breaking changes
3. **Dify 文档无版本号**：文档 URL 中无版本标识，更新是无感知的。建议定期检查文档变化（每周一次），及时更新本地缓存
4. **团队技术栈一致性**：三个方案都推荐 Python（与 Dify 源码同语言、FastMCP 生态成熟、graphon 为 Python 包），避免引入 TypeScript 增加技术栈分裂

### 5.1 功能覆盖度评分依据

| 能力 | 方案 1 | 方案 2 | 方案 3 |
