## 维度 1：方案架构图

### 整体架构

```
用户 (Claude Code)
  │
  ├─ 输入 /dify-workflow → 加载 Skill
  │
  ▼
┌─────────────────────────────────────────────────┐
│  .claude/skills/dify-workflow.md                │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │ §1 指令区：角色定义 + 使用规则             │  │
│  ├───────────────────────────────────────────┤  │
│  │ §2 Schema 区：25+5 节点类型定义           │  │
│  │   (JSON Schema / Markdown 表格)           │  │
│  ├───────────────────────────────────────────┤  │
│  │ §3 语法规则区：变量引用 / 连线 / 条件分支 │  │
│  ├───────────────────────────────────────────┤  │
│  │ §4 模板区：常用工作流 DSL 模板            │  │
│  ├───────────────────────────────────────────┤  │
│  │ §5 校验规则区：DSL 结构校验 checklist     │  │
│  └───────────────────────────────────────────┘  │
│                                                 │
│  §6 llms.txt 入口 URL（实时抓取触发点）          │
└─────────────────────────────────────────────────┘
  │
  │ 当 Skill 中知识不足时
  ├─ WebFetch(llms.txt) → 获取文档索引
  ├─ WebFetch(具体页面) → 按需补充
  │
  ▼
Claude 生成 Dify DSL YAML
```

### 用户交互流程

```
1. 用户在项目中首次使用
   └─ git clone → .claude/skills/dify-workflow.md 自动可用

2. 用户输入 /dify-workflow
   └─ Skill 内容加载到 Claude context（~8-15K tokens）

3. 用户描述需求："帮我建一个客服问答工作流"
   └─ Claude 基于 Skill 中的 Schema + 模板 + 语法规则生成 DSL

4. 知识不足时（如边缘节点配置）
   └─ Claude 自动 WebFetch llms.txt → 获取文档入口
   └─ WebFetch 具体页面 → 补充细节
   └─ 继续生成

5. 输出：完整的 Dify DSL YAML 文件
```

### 知识组织方式

```
dify-workflow.md（单文件，约 8-15K tokens）
├── 元信息：版本、适用 Dify 版本、更新日期
├── 角色指令：你是 Dify 工作流开发助手...
├── Schema 参考：
│   ├── 节点类型速查表（Markdown 表格）
│   ├── 每个节点的字段定义（精简 JSON Schema）
│   └── 节点间连线规则
├── 语法规则：
│   ├── 变量引用：{{#node_id.variable#}}
│   ├── 条件分支：if/elif/else 映射
│   └── 循环/迭代：Iteration 节点用法
├── 模板库：
│   ├── 简单问答：Start → LLM → Answer
│   ├── RAG 检索：Start → Knowledge → LLM → Answer
│   ├── 多轮客服：Start → IF → LLM/HTTP → Answer
│   └── 数据处理：Start → Code → Template → Answer
├── 校验规则：DSL 结构完整性 checklist
└── llms.txt 入口：https://docs.dify.ai/llms.txt
```

## 维度 2：核心能力清单

| # | 功能 | 成熟度 | 说明 |
|---|------|--------|------|
| 1 | 生成符合 Dify 规范的 DSL YAML | **可用** | 25 个内置节点的 Schema 内嵌，Claude 可直接生成 |
| 2 | 变量引用语法指导 | **可用** | `{{#node_id.variable#}}` 等语法规则可完整编码为 Markdown |
| 3 | 工作流模板提供 | **可用** | 5-8 个典型模板，覆盖 80% 使用场景 |
| 4 | DSL 结构校验 | **可用** | checklist 形式，Claude 自检，非程序化校验 |
| 5 | 实时文档补充 | **有限** | 通过 WebFetch(llms.txt) 获取入口，按需抓取具体页面 |
| 6 | 边缘节点深度查询 | **有限** | Skill 内未覆盖的节点，依赖 llms.txt 实时抓取 |
| 7 | 向量语义搜索 | **不可用** | Skill 无检索能力，只能靠 context 内知识 + 实时抓取 |
| 8 | FTS5 全文搜索 | **不可用** | 同上 |
| 9 | 多客户端支持 | **不可用** | 仅 Claude Code 可用，Cursor/Windsurf 等不支持 Skill 格式 |
| 10 | 动态知识更新 | **有限** | 需手动更新 Skill 文件并 git push，无自动同步 |
| 11 | Schema 版本追踪 | **有限** | 需人工维护，无自动 diff 机制 |
| 12 | 节点配置补全提示 | **可用** | Schema 内嵌后，Claude 可根据字段定义给出配置建议 |

**诚实评估**：Skill 方案覆盖了"生成一个能用的 Dify 工作流 DSL"这个核心需求。它做不到的是"在海量文档中精准检索"和"多客户端通用"。但问题是——我们真的需要这些吗？

## 维度 3：数据/接口模型

### 知识格式

| 内容 | 格式 | 理由 |
|------|------|------|
| 节点 Schema | Markdown 表格 + 精简 JSON | Claude 对 Markdown 理解最好；JSON 保真度高 |
| 语法规则 | 自然语言 + 代码示例 | Claude 需要规则 + 示例才能准确应用 |
| 工作流模板 | YAML 代码块 | 直接可复制的 DSL 片段 |
| 校验规则 | Markdown checklist | Claude 可逐项自检 |

### 更新机制

```
手动更新流程（成本极低）：
1. Dify 版本发布
2. 维护者 diff 新旧文档 / Schema
3. 更新 .claude/skills/dify-workflow.md
4. git commit + push
5. 所有用户 git pull 即生效

自动更新流程（可选增强）：
1. CI 脚本定期抓取 llms.txt
2. 与本地 Skill 内容 diff
3. 生成 PR 供人工审核
4. merge 后自动分发
```

### 分发方式

```
方式 1（推荐）：项目内 .claude/skills/ 目录
  - git clone 即可用
  - 版本与项目绑定
  - 零额外部署

方式 2：独立 Skill 仓库
  - 用户手动复制到自己项目
  - 适合跨项目复用

方式 3：npm/pip 包（不推荐）
  - Skill 是纯 Markdown，用包管理器分发是过度工程化
```

### 接口模型

Skill 方案没有传统意义上的"接口"。它的"接口"是：

- **输入**：用户在 Claude Code 中输入 `/dify-workflow` + 自然语言需求
- **输出**：Claude 生成的 Dify DSL YAML
- **补充通道**：Claude 自主调用 WebFetch 访问 llms.txt

这个模型的优雅之处在于：**它不引入任何新的运行时依赖**。Skill 文件就是知识，Claude 就是引擎，WebFetch 就是补充通道。

## 维度 4：扩展点

### 良好扩展（顺势而为）

| 扩展方向 | 难度 | 说明 |
|----------|------|------|
| 增加新节点 Schema | **极低** | 往 Markdown 文件追加一个表格 + JSON 块 |
| 增加工作流模板 | **极低** | 追加 YAML 代码块 |
| 增加校验规则 | **极低** | 追加 checklist 项 |
| 支持多语言提示 | **低** | 加一个 §语言切换 段落 |
| 增加 llms.txt 页面索引 | **低** | 追加 URL 列表 |

### 难扩展（需要额外工程）

| 扩展方向 | 难度 | 说明 |
|----------|------|------|
| 支持 Cursor/Windsurf | **高** | 需要为每个 IDE 写不同格式的规则文件，且功能不等价 |
| 程序化 DSL 校验 | **高** | Skill 内只能做 checklist，真正的 AST 校验需要代码 |
| 跨版本 Schema 管理 | **中** | 需要维护多个版本的 Skill 文件或用条件逻辑 |
| 大规模文档检索 | **中** | 需要依赖 WebFetch + llms.txt，延迟和可靠性受限 |

### 反扩展点（明确不支持）

| 方向 | 原因 |
|------|------|
| 向量语义搜索 | Skill 没有计算能力，无法做 embedding |
| 实时 Schema diff | 无法运行程序，只能人工维护 |
| 多租户/权限控制 | Skill 是静态文件，无运行时 |
| 使用统计/遥测 | 无服务端，无法收集数据 |

## 维度 5：开发成本

### 人天估算

| 任务 | 人天 | 说明 |
|------|------|------|
| 提取 25+5 节点 Schema | 2-3 | 一次性 pip install graphon，脚本提取，整理为 Markdown |
| 编写语法规则文档 | 1-2 | 基于 Dify 官方文档整理变量引用、连线规则 |
| 编写 5-8 个工作流模板 | 1-2 | 手工编写典型场景的 DSL YAML |
| 编写校验 checklist | 0.5 | 基于 DSL 结构整理完整性检查项 |
| 整理 llms.txt 入口 + 测试 WebFetch | 0.5-1 | 确认 llms.txt 可访问，测试抓取流程 |
| 集成测试 + 迭代 | 1-2 | 实际使用生成 3-5 个工作流，修正 Skill 内容 |
| **总计** | **6-10.5** | |

### 与 MCP Server 方案对比

| 维度 | Skill 方案 | MCP Server 方案 |
|------|-----------|----------------|
| 开发人天 | 6-10.5 | 22-30 |
| 学习曲线 | Markdown 编写，零编程 | Python + FastMCP + FAISS + FTS5 |
| 维护成本 | 极低，Markdown 文件 | 中等，服务进程 + 索引重建 |
| 部署成本 | 零（git 即部署） | 需要安装依赖、启动服务 |
| 运行时依赖 | 零 | Python 进程常驻 |
| 故障点 | 无（文件不存在时降级为普通对话） | 服务崩溃 / 索引损坏 / 依赖冲突 |

**关键论点**：Skill 方案的开发成本是 MCP Server 的 **1/3 到 1/4**。这不是"便宜一点"，而是"一个周末能做完 vs 需要一个完整迭代"的区别。

### 学习曲线

- Skill 方案：会写 Markdown 就能维护。不需要会 Python，不需要理解向量数据库，不需要知道什么是 MCP 协议。
- MCP Server 方案：需要 Python 开发能力、FastMCP 框架理解、FAISS 索引调优、FTS5 配置、进程管理。

对于一个"Dify 工作流辅助工具"的维护者来说，Skill 方案的技能要求是合理的；MCP Server 方案的技能要求是过度的。

## 维度 6：致命缺陷自述（强制）

> 自报缺陷永远比被对方挖出更好。以下是方案 2 的三个最大缺陷，每个都附证据。

### 缺陷 1：无动态检索能力，大文档场景下覆盖率受限

**问题**：Skill 是静态 Markdown，无法做向量语义搜索或全文搜索。当 Dify 文档增长到数百页，Skill 内嵌的知识只能覆盖核心部分（约 60-70%），边缘场景需要依赖 WebFetch 实时抓取。

**证据**：
- Dify 文档约 100-200 页，Skill 文件控制在 15K tokens 以内才能保证加载速度和 context 留余。这意味着只能内嵌约 30-40 页的精简内容。
- WebFetch 依赖 llms.txt 索引质量。如果 llms.txt 不完整或页面结构变化，抓取会失败。
- 实际测试中，对于"HTTP 请求节点的高级认证配置"这类边缘问题，Skill 内大概率没有覆盖，需要实时抓取——而实时抓取有延迟（2-5 秒/页面）且可能失败。

**严重程度**：中。核心场景（80%）不受影响，但边缘场景的用户体验不一致。

### 缺陷 2：仅支持 Claude Code，无法服务其他 AI 编程工具

**问题**：Claude Code Skill 是 Claude Code 独有的格式。Cursor、Windsurf、GitHub Copilot 等工具不识别 `.claude/skills/` 目录。

**证据**：
- Cursor 使用 `.cursorrules` 文件，格式和能力与 Skill 不同。
- Windsurf 使用自己的规则系统。
- 如果团队中有人用 Cursor，有人用 Claude Code，Skill 方案无法统一覆盖。
- MCP 协议是跨工具标准，被 Cursor、Windsurf、Claude Desktop 等多个客户端支持。

**严重程度**：高（如果团队有多工具需求）/ 低（如果团队统一使用 Claude Code）。当前项目的定位是"面向 Claude Code 编程智能体"，所以这个缺陷在当前范围内是**可接受的**。

### 缺陷 3：知识更新依赖人工维护，存在版本滞后风险

**问题**：Skill 文件中的 Schema 和文档知识需要人工更新。Dify 版本迭代时，如果维护者没有及时更新 Skill，用户会基于过时信息生成 DSL。

**证据**：
- Dify 的迭代速度较快（GitHub 显示近 3 个月有 500+ commits），节点 Schema 可能随版本变化。
- 目前没有自动化的 Schema diff 机制。维护者需要手动对比新旧版本的差异。
- 与 MCP Server 方案对比：MCP Server 可以通过 graphon 运行时依赖自动获取最新 Schema，而 Skill 方案需要人工介入。

**严重程度**：中。可以通过 CI 自动化脚本缓解（定期检查 llms.txt 变化并生成 PR），但无法完全消除滞后。

## 维度 7：与其他方案的组合可行性

### vs 方案 1（全功能 MCP Server）

**组合策略：Skill 做 MVP，后续按需升级为 MCP Server**

这是最务实的演进路径：

```
Phase 1（当前）：Skill 方案
  └─ 6-10.5 人天，验证核心需求是否被满足
  └─ 收集用户反馈：哪些场景 Skill 搞不定？

Phase 2（按需）：如果 Skill 不够用
  └─ 基于 Phase 1 的反馈，精确知道 MCP Server 需要解决什么问题
  └─ 避免"预设需求"导致的过度工程化
```

**关键论点**：MCP Server 方案需要 22-30 人天，而我们**还不确定用户是否需要**向量搜索、FTS5、多客户端支持。先用 Skill 方案（6-10.5 人天）验证需求，再决定是否投入 MCP Server，这是**风险最低**的策略。

**升级成本评估**：
- Skill → MCP Server 不是"推倒重来"，而是"知识迁移"。Skill 中的 Schema、模板、规则都可以复用到 MCP Server 的知识库中。
- 预计升级成本：在 Skill 基础上增加 15-20 人天（与从零开发 MCP Server 的 22-30 人天相比，节省了前期的 Schema 提取和知识整理工作）。

### vs 方案 3（折中 MCP Server）

**直接对比**：

| 维度 | 方案 2（Skill） | 方案 3（折中 MCP） |
|------|----------------|-------------------|
| 开发成本 | 6-10.5 人天 | 15-20 人天（估） |
| 运行时依赖 | 零 | Python 进程 |
| 多客户端支持 | 不支持 | 支持（MCP 协议） |
| 动态检索 | 无（WebFetch 补充） | llms.txt 轻量检索 |
| 部署复杂度 | git clone | pip install + 启动服务 |

**方案 3 的尴尬之处**：它试图在"MCP Server 的框架"内做简化，但保留了 MCP Server 最大的成本来源——Python 服务进程。如果用户不需要多客户端支持，方案 3 就是"为了用 MCP 协议而多付 10 个人天"。

**方案 2 的优势**：如果目标用户群确实是 Claude Code 用户，方案 3 的多客户端优势毫无价值，而方案 2 的零部署优势是实实在在的。

## 总结（≤ 500 字）

**为什么选方案 2 是最优解。**

方案 2 的核心主张是：**在验证阶段，极简方案可能就够了，MCP Server 是过度工程化。**

这个主张建立在三个事实之上：

**第一，核心需求被满足。** Dify 工作流 DSL 生成的核心依赖是：节点 Schema（25+5 个）、变量引用语法规则、典型工作流模板、DSL 校验规则。这些内容可以完整编码到一个 15K tokens 的 Markdown 文件中。yzmw123/dify-workflow-dsl-skill 已经证明了这个路径的可行性。

**第二，开发成本是 MCP Server 的 1/3 到 1/4。** 6-10.5 人天 vs 22-30 人天。这不是"便宜一点"，而是"一个周末能做完 vs 需要一个完整迭代"的区别。节省的时间可以用来打磨 Skill 内容的质量，而不是搭建基础设施。

**第三，升级路径清晰。** Skill 做 MVP，收集用户反馈，按需升级为 MCP Server。Skill 中的 Schema、模板、规则都可以复用。这不是"选了 Skill 就走不了 MCP"，而是"先用最低成本验证，再决定是否投入更多"。

方案 2 的缺陷是真实的：无动态检索、仅支持 Claude Code、知识更新依赖人工。但这些缺陷在当前阶段都是**可接受的**——因为我们还没有证据表明用户真的需要向量搜索、真的需要多客户端支持、真的需要实时 Schema 同步。

**正确的工程决策不是"哪个方案功能最全"，而是"哪个方案用最低成本验证核心假设"。** 方案 2 就是那个方案。

如果未来用户反馈表明 Skill 不够用，我们随时可以升级到 MCP Server——而且那时我们已经有了经过验证的知识资产（Schema、模板、规则），升级成本远低于从零开始。
