# 候选方案分配（Phase 0）

总候选数 N = 3（架构方向选型，非开源项目选型）
项目定位：面向 Claude Code 编程智能体的 Dify 工作流开发辅助工具

## 本次适配的 7 维度含义

| # | 维度 | 本次具体含义 |
|---|------|------------|
| 1 | 方案架构图 | 该方案的技术架构、模块组成、部署方式 |
| 2 | 核心能力清单 | 该方案能实现什么功能、不能实现什么 |
| 3 | 数据/接口模型 | 数据存储方式、对外接口格式、依赖链 |
| 4 | 扩展点 | 后续加功能、换组件、支持多客户端的难度 |
| 5 | 开发成本 | 人天估算、学习曲线、维护成本 |
| 6 | ⭐ 致命缺陷自述 | 该方案最大的 3 个缺陷（强制） |
| 7 | 与其他方案的组合可行性 | 能否与其他方案组合（如 Skill 做 MVP + MCP Server 做正式版） |

## 分配结果

### 代言人 A owns：方案 1「全功能 MCP Server」
- MCP Server（Python FastMCP）+ 自建 FAISS+FTS5 索引 + graphon 运行时依赖
- 代表"完整独立服务"立场
- 理由：这是调研文档的原始推荐方案，需要有人为其强势辩护

### 代言人 B owns：方案 2「极简 Skill」
- Claude Code Skill（纯 Markdown）+ llms.txt 实时抓取 + 一次性提取 Schema 内嵌
- 代表"极简轻量"立场
- 理由：yzmw123/dify-workflow-dsl-skill 已有先例，可能是被忽视的更优路径

### 代言人 C owns：方案 3「折中 MCP Server」
- MCP Server（Python FastMCP）+ llms.txt 轻量检索（不建向量索引）+ 一次性提取 Schema 本地维护
- 代表"折中务实"立场
- 理由：在 MCP Server 框架内做最大简化，验证是否需要向量检索和 graphon 运行时依赖

### 红队
固定 1 个。立场：反对所有方案，主张"全自建"或"换思路"。

### 集成评估师
固定 1 个。立场：中立，评估各方案的工程可行性和组合方案。

## 总体策略

这 3 个决策点是**级联依赖**的：
- 决策点 1（MCP Server vs Skill）决定整体架构方向
- 决策点 2（检索方案）和决策点 3（Schema 策略）在 MCP Server 路线下才有意义

但 Skill 路线也不能跳过——如果 Skill 够用，决策点 2/3 都不需要。所以每个代言人都需要完整评估自己的方案是否真的能解决问题。
