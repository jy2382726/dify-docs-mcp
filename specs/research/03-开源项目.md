# 主题 3：开源项目调研

## 1. 调研概述

本文档对 GitHub 上与「Dify 工作流 MCP 服务」同类的开源项目进行系统调研。调研覆盖三大类：(A) Dify 生态 MCP 服务器、(B) 文档/知识库查询 MCP 服务器、(C) MCP 框架与校验工具。目标是为团队内部的「文档查询 + 节点 Schema 查询 + 变量引用查询 + DSL 校验 + Dify API 集成」服务找到可复用的开源基础。

**数据来源标注**：`[web-search-prime]` = 网络搜索结果，`[zread]` = 仓库结构分析。单源标注 ⚠️，双路验证标 ✅。

## 3. 生态拼图

### 3.1 能力维度分析

本项目需要 5 项核心能力，现有开源生态的覆盖情况如下：

| 能力维度 | 生态现状 | 最佳候选 |
|----------|----------|----------|
| **MCP Server 框架** | 生态成熟。官方 TS/Python SDK (12k+ stars) + FastMCP (15k+ stars) 提供稳定基础 | ✅ `modelcontextprotocol/typescript-sdk` 或 `PrefectHQ/fastmcp` |
| **文档查询 (RAG)** | 生态丰富但通用化。`docs-mcp-server` 最成熟，`knowledge-rag` 最全面 | ✅ `arabold/docs-mcp-server` (TypeScript, MIT, 语义搜索+chunking) |
| **节点 Schema 查询** | **空白地带**。仅 `n8n-mcp` 对 n8n 做了类似事情，Dify 生态无对应 | ⚠️ `czlonkowski/n8n-mcp` 架构可复用 (SQLite + FTS5 + Schema 存储) |
| **变量引用查询** | **空白地带**。无现成项目 | 需从 0 构建，参考 n8n-mcp 的搜索模式 |
| **DSL 校验** | **空白地带**。`yzmw123/dify-workflow-dsl-skill` 有 DSL 知识但无校验逻辑 | 需从 0 构建，可复用 `Janix-ai/mcp-validator` 的校验思路 |

### 3.2 关键发现

**最相似项目 = n8n-mcp** [zread] ⚠️

`czlonkowski/n8n-mcp` 是与我们需求架构最相似的项目。它解决的问题几乎完全一致，只是目标平台不同（n8n vs Dify）：

- 用 SQLite 存储节点文档、属性、操作的结构化数据
- FTS5 全文搜索支持模糊查询
- 节点 Schema 校验（包括类型版本、操作验证、表达式格式验证、webhook 验证等）
- Workflow 验证（连接验证、空连接检测、错误输出验证等）
- MCP Server 暴露为 tools，支持 stdio 和 HTTP 双模式
- Docker 部署，完善的测试套件

**Dify MCP 生态碎片化**

现有 7 个 Dify MCP 项目，但全部聚焦于「调用 Dify Workflow API」，没有一个做「查询 Dify 文档/节点 Schema/变量引用」。这意味着我们要做的事情在 Dify 生态中是**首创**，无直接 fork 对象。

**DSL Skill 项目值得关注**

`yzmw123/dify-workflow-dsl-skill` 和 `r-hashi01/skills-to-dify-workflow` 虽然是 Claude Code Skill 而非 MCP Server，但它们积累了 Dify DSL 的结构知识（节点类型、图连线规则、DSL 版本差异），这些知识可以提取为我们的 Schema 校验逻辑。

## 4. 推荐组合

### 4.1 推荐策略：「n8n-mcp 架构参考 + FastMCP 框架 + docs-mcp-server 文档层参考」

遵循「架构参考 > 模式借鉴 > 从 0 写」的原则（不直接 fork 任何项目，详见 4.2）：

| 层次 | 推荐方案 | 来源项目 | 集成方式 |
|------|----------|----------|----------|
| **MCP Server 框架** | 使用 Python FastMCP 框架 | `PrefectHQ/fastmcp`（15k stars, Apache-2.0） | pip 依赖引入 |
| **文档查询层** | 参考 docs-mcp-server 的文档抓取+索引+搜索架构 | `arabold/docs-mcp-server`（MIT） | 架构参考，Python 重写 |
| **节点 Schema 存储+查询** | 参考 n8n-mcp 的 SQLite + FTS5 + Schema 模式 | `czlonkowski/n8n-mcp`（MIT） | 架构参考，Python 重写，数据源替换为 graphon |
| **DSL 校验逻辑** | 从 Dify 源码 + DSL Skill 项目提取规则 | `yzmw123/dify-workflow-dsl-skill` + Dify 源码 | 提取 DSL 知识，从 0 写校验引擎 |
| **Dify API 集成** | 参考 zhuzhoulin 的 Dify API 调用模式 | `zhuzhoulin/dify-mcp-server` | 模式参考 |

### 4.2 为什么不直接 fork 某个项目

| 候选 | 不直接 fork 的原因 |
|------|-------------------|
| zhuzhoulin/dify-mcp-server | 仅做 API 桥接，架构太简单，缺少文档/Schema/校验层 |
| n8n-mcp | 目标平台不同（n8n vs Dify），语言不同（TypeScript vs Python），数据层需完全重写。但其 SQLite + FTS5 架构模式值得参考 |
| docs-mcp-server | 面向通用第三方文档，需改造为 Dify 文档专用；语言不同（TypeScript） |
| hjlarry/dify-plugin-mcp_server | 依赖 Dify 插件系统，不能独立部署为 Claude Code 的 MCP Server |

**结论**：所有候选项目都不适合直接 fork。我们的策略是**参考架构模式 + Python 自研**，利用 graphon 包作为 schema 数据源。

### 4.3 最小可行方案（MVP）

**Phase 1 - 基础框架**（1 周）：
- 基于 `PrefectHQ/fastmcp` 搭建 MCP Server 骨架
- 参考 `n8n-mcp` 的 SQLite 数据层设计，创建 Dify 节点 Schema 数据库
- 从 `langgenius/graphon` 包提取 25 个内置节点 schema
- 实现 3 个基础 tools：`search_docs`、`get_node_schema`、`get_variable_ref`

**Phase 2 - 文档+Schema**（2 周）：
- 参考 `docs-mcp-server` 实现 Dify 文档抓取、chunking、索引
- 从 graphon + dify 源码提取完整节点类型定义，填充 Schema 数据库
- 实现 FTS5 全文搜索（参考 n8n-mcp 的搜索模式，用 Python sqlite3 实现）

**Phase 3 - 校验+API**（1 周）：
- 从 `dify-workflow-dsl-skill` 提取 DSL 规则，实现基础校验
- 参考 `zhuzhoulin/dify-mcp-server` 集成 Dify Workflow API 调用
- Docker 部署支持

### 4.4 技术栈建议

- **语言**：Python（Dify 源码同语言，graphon 包为 Python，FastMCP 生态最成熟，团队熟悉度高）
- **MCP 框架**：`PrefectHQ/fastmcp`（Python，15k stars，Apache-2.0）
- **数据存储**：SQLite + FTS5（n8n-mcp 验证过的模式，Python 中同样适用）
- **部署**：stdio（Claude Code 原生支持）+ HTTP（可选，供其他客户端使用）

**为什么选 Python 而非 TypeScript**：
- graphon 包（核心节点 schema 来源）是 Python，可直接 `pip install graphon` 后用 Python AST 解析 entities.py
- FastMCP（Python，15k stars）比官方 TS SDK 更易用，装饰器式 API 开发效率更高
- 与 Dify 源码同语言，便于参考和复用 Dify 的校验逻辑
- n8n-mcp 虽为 TypeScript，但其 SQLite + FTS5 架构模式在 Python 中同样适用（用 `sqlite3` 标准库即可）

## 5. 风险与注意事项

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| n8n-mcp 架构复用成本 | 虽然模式相似，但 n8n 和 Dify 的节点模型差异较大 | 先做 Spike，验证数据层迁移可行性 |
| Dify DSL 规范无官方文档 | Dify 官方讨论区承认「DSL 本质上是前端画布的 JSON 结构」[web-search-prime] ⚠️ | 从 Dify 源码 (`api/core/app/apps/workflow`) 直接提取 |
| 文档抓取的 Dify 版本适配 | Dify 快速迭代，文档可能滞后 | 实现增量更新机制，参考 docs-mcp-server 的更新策略 |
| 小型项目的维护风险 | 多数 Dify MCP 项目 stars < 50，可能停止维护 | 仅拆模块用，不深度依赖 |
