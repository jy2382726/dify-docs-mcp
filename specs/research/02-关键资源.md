### 2B-2. 节点类型的定义文件位置

**关键发现**：Dify 的核心节点类型（LLM、Code、HTTP、If-Else 等）已迁移至独立的 **`graphon` 包**（`langgenius/graphon`，PyPI: `graphon==0.4.0`）。[zread] ✅ 双路验证（源码 node_factory.py + graphon 仓库结构）

**节点注册机制**（`api/core/workflow/node_factory.py`）：
```python
def register_nodes() -> None:
    _import_node_package("graphon.nodes")       # 内置节点来自 graphon
    _import_node_package("core.workflow.nodes")  # Dify 特有扩展节点
```

**graphon 包含 25 个内置节点类型**（`src/graphon/enums.py` 中 `BuiltinNodeTypes`）：[zread] ✅
- `start`、`end`、`answer` — 起止节点
- **`llm`**、**`code`**、**`http-request`** — 处理节点
- **`if-else`**、`iteration`、`iteration-start`、`loop`、`loop-start`、`loop-end` — 流程控制
- `template-transform`、`question-classifier`、`parameter-extractor` — AI 辅助节点
- `document-extractor`、`list-operator`、`human-input` — 数据处理
- `tool`、`variable-aggregator`、`variable-assigner`、`agent`、`knowledge-retrieval`、`datasource`

**当前 `api/core/workflow/nodes/` 目录仅包含 Dify 特有扩展节点**：
- `agent/`、`agent_v2/` — Agent 节点（Dify 特有扩展）
- `datasource/` — 数据源节点
- `knowledge_index/`、`knowledge_retrieval/` — 知识库节点
- `trigger_plugin/`、`trigger_schedule/`、`trigger_webhook/` — 触发器节点

**Schema 提取路径已修正**：核心节点的 Pydantic entities 在 graphon 仓库中（`src/graphon/nodes/*/entities.py`），不在主仓库中。提取方案需同时覆盖两个仓库。

**节点实体定义**：graphon 中每个节点子目录下有 `entities.py`，使用 Pydantic BaseModel 定义输入输出 schema，继承自 `graphon.entities.base_node_data.BaseNodeData`。[zread] ✅

### 2B-3. DSL 文件的解析和校验逻辑

核心文件：**`api/services/app_dsl_service.py`** [web-search-prime]

- 导入逻辑：`_create_or_update_app()` 方法（约 406-532 行）
- DSL 格式：YAML，最大 10MB
- 版本管理：`api/services/dsl_version.py` 负责版本兼容性检查
- 控制器入口：`api/controllers/console/app/app_import.py`（控制台 UI 导入）

**校验逻辑**：DSL 导入时会检查版本兼容性，但**没有独立的校验 API 端点**。校验逻辑嵌入在导入流程中。[web-search-prime]

### 2B-4. 变量引用的解析逻辑

变量引用格式（已从源码确认）：**`{{#node_id.variable#}}`** 是当前活跃的变量引用格式。

**源码证据**：[zread] ✅ 双路验证（前端 VAR_REGEX + 后端变量选择器）
- 前端 `web/config/index.ts` 中的正则：`/\{\{(#[\w-]{1,50}(\.\d+)?(\.[a-z_]\w{0,29}){1,10}#)\}\}/gi`
  匹配格式如 `{{#node_id.output_key#}}`、`{{#sys.query#}}`、`{{#env.api_key#}}`
- 前端变量选择器用数组表示 `["node_id", "output_key"]`，与 `{{#...#}}` 格式互转
- 后端 `system_variables.py` 中 `inject_default_system_variable_mappings` 使用 `f"{node_id}.#sys.query#"` 格式

**特殊变量前缀**（`api/core/workflow/variable_prefixes.py`）：[zread] ✅
- `sys` — 系统变量（query, files, user_id, conversation_id 等，共 16 个）
- `env` — 环境变量（只读全局）
- `conversation` — 会话变量（多轮持久化）
- `rag` — RAG Pipeline 变量

**注意**：`{{variable_name}}`（不含 `#`）格式用于 Agent Prompt 模板中的占位符（如 `{{instruction}}`、`{{tools}}`），不是工作流变量引用。两种格式用途不同，不要混淆。

**模板渲染**：`api/core/workflow/template_rendering.py` 已迁移为 graphon 的 `Jinja2TemplateRenderer` 的薄包装，实际渲染逻辑在 graphon 包中。LLM 节点支持 Jinja2 模板模式（`edition_type: "jinja2"`），变量通过 `{% for %}` 和 `{{ }}` 语法处理。[zread] ✅

### 2B-5. 从源码自动提取节点 Schema 的可行性

**可行，但有难度**：

| 方案 | 难度 | 说明 |
|------|------|------|
| 解析 entities.py | 中 | 每个节点的 `entities.py` 定义了 Pydantic model，可自动提取字段名、类型、默认值 |
| 解析前端 i18n | 低 | `web/i18n/en-US/workflow.ts` 包含节点类型的 UI 文案，可用于补充描述 |
| 解析 OpenAPI spec | 低 | 官方提供 `openapi_workflow.json`，但仅覆盖 Service API，不包含节点内部 schema |
| 导出 DSL 样本逆向 | 中 | 从多个 DSL YAML 文件中逆向推断节点结构 |

**推荐方案**：组合使用 entities.py 解析 + DSL 样本逆向 + 官方文档抓取，三路互补。

## 2C. Dify API

### 2C-1. 获取节点类型列表的 API

**不存在公开 API**。Dify 没有提供"列出所有节点类型"的 REST API 端点。[web-search-prime]

节点类型列表需要从以下来源获取：
- 源码中的 NodeType 枚举（`api/models/workflow.py`）
- 前端 i18n 文件（`web/i18n/en-US/workflow.ts`）
- 官方文档的节点页面列表（`llms.txt`）

### 2C-2. 导入/导出工作流 DSL 的 API

**控制台 API 存在但非公开**：[web-search-prime]

- 导入：`api/controllers/console/app/app_import.py` — 控制台内部使用
- 导出：DSL 导出功能仅通过 UI 操作（Studio 页面或编排页面的"导出 DSL"按钮）
- 内部 API：`api/inner_api/app/dsl.py` — 服务间调用

**Service API（公开）不包含 DSL 导入/导出端点**。GitHub issue #24087 有用户请求通过 API 导入 DSL，但官方尚未提供。[web-search-prime]

### 2C-3. 校验 DSL 合法性的 API

**不存在公开 API**。DSL 校验逻辑嵌入在 `app_dsl_service.py` 的导入流程中，没有独立的校验端点。[web-search-prime]

### 2C-4. 在线运行/测试工作流的 API

**存在，且文档完善**：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/workflows/run` | POST | 执行已发布的工作流 |
| `/workflows/{task_id}/stop` | POST | 停止运行中的工作流 |
| `/workflows/{workflow_run_id}` | GET | 获取运行详情 |
| `/workflows/run`（流式） | POST | SSE 流式返回执行事件 |

支持两种响应模式：`blocking`（阻塞）和 `streaming`（SSE 流式）。[web-search-prime]

### 2C-5. API 认证方式、限频、版本管理

- **认证**：Bearer Token（API Key），放在 `Authorization` 头中 [web-search-prime]
- **API Key 来源**：每个应用独立的 API Key，在 Dify 控制台生成
- **限频**：官方文档未明确说明具体限频策略 [待验证]
- **版本管理**：API 路径为 `/v1/`，当前无 v2；OpenAPI spec 随版本更新
- **OpenAPI 规范**：官方提供 JSON 格式的 OpenAPI spec，可直接用于代码生成 [web-reader]

**OpenAPI 规范 URL**：
- `https://docs.dify.ai/zh/api-reference/openapi_workflow.json`（二进制 JSON，需直接下载）[web-reader]
- `https://docs.dify.ai/zh/api-reference/openapi_chatflow.json`
- `https://docs.dify.ai/zh/api-reference/openapi_knowledge.json`

## 推荐路径

**第一阶段（P0，立即启动）**：
1. 抓取 `llms.txt` 索引，建立文档 URL 列表
2. 批量抓取 20 个节点文档页面，缓存为本地 Markdown
3. 从 `langgenius/graphon` 包提取 25 个内置节点 schema（`src/graphon/nodes/*/entities.py`）
4. 从 `langgenius/dify` 提取扩展节点 schema（Agent/Knowledge/Trigger/Datasource）
5. 从前端 `web/config/index.ts` 的 VAR_REGEX 提取变量引用正则规范
6. 从 `variable_prefixes.py` 和 `system_variables.py` 提取变量前缀和系统变量定义

**第二阶段（P1，核心功能就绪后）**：
1. 下载 OpenAPI workflow JSON，解析为 API 端点参考
2. 从 `app_dsl_service.py` 逆向 DSL 格式规范
3. 编写内部 Dify 工作流开发规范文档

**第三阶段（P2，完善阶段）**：
1. 克隆 Awesome-Dify-Workflow，提取 DSL 样本
2. 从前端 i18n 文件补充节点描述
3. 建立 DSL 校验规则库

**为什么不用 X**：
- **不直接解析前端代码获取节点 schema**：前端代码（React/TypeScript）的节点定义与后端 Pydantic model 存在差异，以后端为准更可靠
- **不依赖控制台 API 做 DSL 导入导出**：控制台 API 需要 session 认证，不适合 MCP 服务场景；应专注于 Service API
- **不维护独立的文档 fork**：直接抓取官方文档并缓存，避免维护成本；文档更新时重新抓取即可
