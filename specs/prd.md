## 2. 项目背景与目标

### 2.1 背景

团队使用 Dify 平台开发 AI 工作流，但编写 Dify DSL YAML 文件存在以下痛点：

- **手动编写易错**：变量引用格式（`{{#node_id.field#}}`）容易写错，节点配置字段遗漏常见
- **查文档效率低**：Dify 文档分散在 20+ 页面，开发者需要在文档站点和编辑器之间频繁切换
- **无校验手段**：Dify 没有提供独立的 DSL 校验 API，错误只能在导入时才发现

团队已使用 Claude Code 作为 AI 编程智能体，但 Claude Code 缺乏 Dify 工作流的领域知识——不知道节点类型、变量语法、DSL 结构。需要一个工具让 Claude Code 具备 Dify 工作流开发能力。

**市场空白**：在「AI 辅助工作流开发」赛道，n8n 有成熟的 MCP Server 实现（20+ tools），但 Dify、Coze、FastGPT 均无类似产品。（来源：`specs/research/01-产品形态.md`）

### 2.2 目标（Goals）

- **业务目标**：让团队用 Claude Code 高效开发 Dify 工作流，降低 DSL 编写门槛和错误率
- **产品目标**：交付一个工具，使 Claude Code 能根据自然语言描述生成可直接导入 Dify 的 DSL YAML 文件
- **学习目标**：验证「AI 辅助 Dify 工作流开发」这一需求是否真实存在，以及 Skill 形态的覆盖度是否足够

### 2.3 非目标（Non-Goals）

- 不做可视化工作流编辑器或 Web UI
- 不做工作流运行时管理（部署/监控/回滚）
- 不做 Dify 平台的替代品
- 不做非 Claude Code 客户端的支持（后续迭代再考虑）
- 不做本地离线文档搜索（依赖外部文档服务）
- 不做用户认证/权限系统

### 2.4 战略对齐

本项目验证「AI 辅助低代码平台工作流开发」的产品方向。如果验证成功，可扩展到其他低代码平台的工作流开发场景。

## 3. 目标用户与画像

### 3.1 主要画像：Dify 工作流开发者（使用 Claude Code）

- **背景**：技术团队成员，熟悉 Dify 平台，了解工作流节点和 DSL 结构
- **目标**：用自然语言描述需求，让 Claude Code 生成可导入的工作流，减少手动编写 DSL 的时间和错误
- **痛点**：
  - 手写 DSL YAML 容易出错（变量引用格式、节点配置遗漏）
  - 查文档效率低（需要在文档站点和编辑器之间切换）
  - 新节点类型的学习成本高（每次 Dify 更新可能引入新节点）
- **使用频次**：每次开发 Dify 工作流时使用（预计每周 2-5 次）
- **设备**：PC（终端 + Claude Code）

### 3.2 反画像（明确不服务谁）

- **非技术用户**：需要可视化拖拽编辑器，本工具是 CLI/终端交互
- **非 Claude Code 用户**：使用 Cursor/Windsurf 的团队，本阶段不支持
- **运维角色**：需要管理工作流生命周期（部署/监控/回滚），本工具只管生成不管运维

## 4. 用户故事

### US-01：自然语言生成工作流

作为 Dify 工作流开发者
我希望用自然语言描述一个工作流需求
以便 Claude Code 生成可导入的 DSL YAML，不需要手写复杂的节点配置和变量引用

**触发情境**：开发者需要创建一个新的 Dify 工作流，在 Claude Code 中描述需求
**期望结果**：Claude Code 输出完整的 DSL YAML 文件，可直接导入 Dify 平台

### US-02：DSL 校验与修复

作为 Dify 工作流开发者
我希望 Claude Code 在生成 DSL 后自动校验并修复常见错误
以便导入 Dify 时不会遇到格式问题

**触发情境**：Claude Code 刚生成了一个 DSL YAML，准备交付给用户
**期望结果**：校验通过的 DSL 直接交付；发现错误时自动修复后重新交付

### US-03：查询节点文档和变量语法

作为 Dify 工作流开发者
我希望 Claude Code 能查询特定节点的配置方式和变量引用语法
以便我能正确配置复杂节点（如 LLM 的 Context RAG、Jinja2 模板）

**触发情境**：开发者在编写工作流时遇到不确定的节点配置
**期望结果**：Claude Code 通过文档搜索返回准确的配置说明和示例

## 6. 详细功能描述

### F1 文档语义搜索

**触发**：Claude Code 在生成工作流过程中需要查询 Dify 文档

**输入**：
- `query`：搜索关键词或自然语言问题（必填）
- `version`：Dify 版本号（选填，默认最新）
- `language`：文档语言（选填，默认 en）

**核心流程**：
1. Claude Code 调用外部文档搜索服务
2. 服务返回相关文档片段（标题 + 内容 + 来源 URL）
3. Claude Code 从结果中提取有用信息

**输出**：
- 相关文档片段列表（Markdown 格式）

**边界条件**：
- 文档服务不可用 → 降级为使用本地 Skill 中的 Schema 和模板
- 搜索结果不精准 → 使用文件系统命令直接读取特定页面
- 查询超出速率限制 → 等待后重试

**性能要求**：响应时间 < 3 秒（P95）

### F2 节点 Schema 参考

**触发**：Claude Code 需要了解某个节点类型的配置方式

**输入**：
- 节点类型名（如 `llm`、`if-else`、`code`）

**核心流程**：
1. Claude Code 从本地 Skill 文件中查找对应节点的 Schema
2. 返回完整的参数定义、类型约束、配置示例

**输出**：
- 节点的完整配置 Schema（Markdown 格式，含 YAML 示例）

**边界条件**：
- 节点类型不在 25+ 列表中 → 通过文档搜索服务查询该节点文档，自行推导配置
- 用户的 Dify 版本与 Schema 版本不一致 → 在 Skill 中标注版本号，差异部分通过文档搜索补充

**覆盖范围**：25 种内置节点 + 5 种扩展节点（Agent、Knowledge、Trigger 等）

### F3 变量引用语法规则

**触发**：Claude Code 在生成 DSL 时需要使用变量引用

**输入**：无（规则文档，Claude Code 按需读取）

**核心流程**：
1. Claude Code 从 Skill 文件中读取变量语法规则
2. 按规则生成正确的变量引用格式

**输出**：
- 变量引用格式说明（`{{#node_id.field#}}` 工作流变量 vs `{{variable}}` Prompt 占位符）
- 变量前缀说明（sys / env / conversation / rag）
- 系统变量列表（16 个：query, files, user_id 等）

**边界条件**：
- 用户不清楚两种格式的区别 → Skill 中明确区分并给出使用场景

### F4 DSL 校验 checklist

**触发**：Claude Code 生成 DSL YAML 后，在交付给用户之前

**输入**：
- 生成的 DSL YAML 内容

**核心流程**：
1. Claude Code 按 checklist 逐项检查 DSL
2. 发现错误 → 自动修复 → 重新输出
3. 无法自动修复 → 返回 warning 级提示

**输出**：
- 校验通过：直接交付 DSL
- 校验不通过：修复后的 DSL + 遗留问题的 warning 列表

**校验项**（15+ 条规则）：
- 必填字段完整性（app、workflow、graph、nodes、edges）
- 节点 ID 唯一性
- 边引用的节点必须存在
- 变量引用格式正确（`{{#node_id.field#}}`）
- 变量引用的上游节点必须在 DAG 中可达
- 起止节点完整性（start、end）

**边界条件**：
- checklist 无法覆盖的语义错误 → 返回 warning，提示用户手动检查
- 校验脚本执行失败 → 降级为纯 LLM 自检

### F5 工作流模板库

**触发**：用户请求创建工作流，且存在匹配的模板

**输入**：
- 用户的需求描述

**核心流程**：
1. Claude Code 从 Skill 的模板库中查找匹配模板
2. 基于模板调整节点配置以满足用户需求
3. 输出定制化后的 DSL YAML

**输出**：
- 可直接导入的 DSL YAML 文件

**模板覆盖**（7 个）：
- 简单 LLM 对话
- 条件分支（If-Else）
- HTTP 请求 + 代码处理
- Chatflow 多轮对话
- RAG 知识库检索（从第三方移植）
- 迭代处理列表（从第三方移植）
- 错误处理模式（从第三方移植）

**边界条件**：
- 没有匹配的模板 → Claude Code 基于 Schema 和文档从零生成
- 模板与用户 Dify 版本不兼容 → 通过文档搜索服务查最新配置

### F6 节点输出字段参考

**触发**：Claude Code 需要引用某个节点的输出变量

**输入**：
- 节点类型名

**核心流程**：
1. Claude Code 从 Skill 文件中查找该节点的输出字段列表
2. 使用正确的格式生成变量引用（`{{#node_id.output_field#}}`）

**输出**：
- 节点的输出字段列表（如 LLM 节点：`text`、`usage`、`finish_reason`）

### F7 文档服务使用指引

**触发**：Claude Code 需要决定是否调用文档搜索服务

**输入**：无（指引文档）

**核心流程**：
1. Claude Code 按指引判断何时调用文档搜索（遇到不确定的节点配置、需要最新文档、Skill 中没有的信息）
2. 使用正确的工具和参数调用

**输出**：
- 何时调用的判断规则
- 两个搜索工具的使用方式和参数说明

## 7. 非功能需求（NFR）

### 7.1 性能

| 指标 | 目标值 | 依据 |
|------|--------|------|
| 文档搜索响应时间 | < 3 秒（P95） | 外部服务端点实测 |
| 本地 Skill 文件加载 | < 1 秒 | 纯 Markdown 文件，无网络依赖 |
| 端到端工作流生成时间 | < 30 秒 | 从需求描述到 DSL 输出，包含 1-2 次文档搜索 |

### 7.2 可用性

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 外部文档服务可用性 | 99.5% | 外部依赖，不可控但有降级方案 |
| 本地 Skill 功能可用性 | 100% | 本地文件，无外部依赖 |
| 降级模式 | 纯 Skill 模式 | 文档服务不可用时，Schema/模板/校验仍可用 |

### 7.3 安全

- 不存储用户数据（Skill 本地运行，文档搜索查询公开文档）
- API 密钥通过配置文件管理，不在工具参数中暴露

### 7.4 合规

- 查询内容为公开文档，无数据合规风险
- 不涉及用户隐私数据

### 7.5 容量

| 指标 | 目标值 | 说明 |
|------|--------|------|
| Phase 1 并发用户 | 无上限 | 每人独立实例，无共享资源 |
| Skill 内容大小 | < 15K token | yzmw123 原始 Skill 约 8-15K token |

## 9. 验收标准

### AC-F1-01：文档搜索返回相关结果

**Given** 用户已配置文档搜索服务
**When** Claude Code 搜索 "LLM 节点 Context RAG 配置"
**Then** 返回 LLM 节点文档中关于 Context RAG 的完整配置说明

### AC-F1-02：文档服务不可用时降级

**Given** 文档搜索服务不可用
**When** Claude Code 尝试搜索文档
**Then** 降级为使用本地 Skill 中的 Schema 信息，不报错

### AC-F2-01：覆盖 25+ 节点类型

**Given** Skill 已加载
**When** 查询任意一种内置节点类型（如 `llm`、`if-else`、`code`、`iteration`）
**Then** 返回该节点的完整 Schema（参数定义 + 类型约束 + 配置示例）

### AC-US1-01：自然语言生成可导入的工作流

**Given** 用户已配置文档搜索服务和 Skill
**When** 用户描述 "创建一个工作流：接收用户问题 → 搜索知识库 → 用 LLM 回答"
**Then** Claude Code 生成完整的 DSL YAML，包含 start → knowledge-retrieval → llm → end 节点，所有节点 data 字段完整，edges 正确连接，变量引用格式为 `{{#node_id.field#}}`

### AC-US2-01：DSL 校验发现并修复错误

**Given** Claude Code 已生成一个 DSL YAML
**When** Skill 中的校验 checklist 被触发
**Then** 检查项包括：必填字段完整性、节点 ID 唯一性、边引用的节点存在、变量引用格式正确；发现错误时自动修复并重新输出

### AC-US3-01：查询复杂节点配置

**Given** 用户询问 "LLM 节点怎么配置 Jinja2 模板模式？"
**When** Claude Code 调用文档搜索服务
**Then** 返回 LLM 节点文档中关于 Jinja2 模板的完整配置说明，包括 `edition_type` 字段和模板语法

- 需要程序化 DSL 校验
- 团队中有人用 Cursor/Windsurf
- 需要直接从 Claude Code 执行工作流（导入失败自动调试）

## 11. 度量指标

### 11.1 北极星指标（North Star）

**生成的 Dify 工作流 DSL 首次导入成功率**

- 定义：Claude Code 生成的 DSL YAML 文件在 Dify 平台上首次导入即成功的比例
- 目标：≥ 80%
- 测量方式：团队试用期间记录每次导入结果

### 11.2 关键指标

**Activation（激活）**
- Skill 安装后 24h 内成功生成 ≥ 1 个工作流的用户占比 ≥ 80%

**Retention（留存）**
- 试用 2 周后仍在使用的用户占比 ≥ 60%

**Efficiency（效率）**
- 使用工具生成工作流 vs 手动编写的时间比 ≤ 1:3（生成时间是手动的 1/3 以下）

### 11.3 反指标（防止"指标作弊"）

- 生成的工作流复杂度不能低于手写工作流（防止只生成 trivial 工作流刷成功率）
- 用户需要修改生成结果的比例应 < 30%（防止生成但不能用）

### 11.4 度量频率

- 每次使用记录导入结果（成功/失败/需修改）
- 试用期结束时汇总分析
